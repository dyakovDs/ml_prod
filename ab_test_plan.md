# A/B-тест: Model v1 vs Model v2

## 1. Постановка задачи

**Контекст.** В production работает модель v1 (LogisticRegression, ROC-AUC 0.708). Разработана challenger-модель v2 (RandomForestClassifier, ROC-AUC 0.757). Цель A/B-теста — подтвердить, что v2 статистически значимо лучше v1 по бизнес-метрикам, прежде чем полностью переключить трафик.

**Гипотезы:**
- H0: доля верно классифицированных дефолтов (Recall класса 1) у v1 и v2 одинакова.
- H1: v2 обеспечивает более высокий Recall класса 1.

---

## 2. Разделение трафика

| Группа | Модель | Доля трафика |
|---|---|---|
| Контроль (A) | v1 — LogisticRegression | 50% |
| Тест (B) | v2 — RandomForestClassifier | 50% |

**Механизм маршрутизации.** Текущий API поддерживает выбор модели через query-параметр `?version=v1|v2`. В production-сценарии роутинг реализуется на уровне API-gateway: входящие запросы случайно (на основе хэша `client_id` или UUID сессии) направляются на одну из версий — это гарантирует детерминированность (один клиент всегда попадает в одну группу) и исключает утечку между группами.

**Продолжительность теста.** Минимум **4 недели** — достаточно для сбора ~6 000 новых наблюдений (при ~200 запросах/сутки) и получения статистической мощности >= 0.8 при mde = 0.05 (5 п.п. по Recall).

---

## 3. Метрики

### Основная метрика — F1-score (класс «дефолт»)

F1 выбран как основная метрика, поскольку датасет несбалансирован (~22% дефолтов) и одинаково важно не пропускать реальные дефолты (Recall) и не отклонять хороших клиентов (Precision).

$$F1 = \frac{2 \cdot \text{Precision} \cdot \text{Recall}}{\text{Precision} + \text{Recall}}$$

### Дополнительная метрика — Recall (класс «дефолт»)

Recall приоритетнее Precision в кредитном скоринге: пропущенный дефолт (FN) стоит банку значительно дороже, чем необоснованный отказ (FP). Цель: чтобы v2 имела Recall ≥ v1 при сравнимом Precision.

### Бизнес-метрики (см. README)

- **Expected Loss Reduction**: снижение суммарных ожидаемых потерь по False Negatives.
- **Approval Rate при фиксированном риске**: доля одобренных заявок при ограничении доли дефолтов <= 10%.

## 4. Статистический анализ

### Метод: z-тест для разности двух пропорций

Recall — это пропорция: `TP / (TP + FN)`. Для сравнения двух пропорций применяется двусторонний z-тест.

**Формулы:**

Пусть $p_A$ — Recall группы A (v1), $p_B$ — Recall группы B (v2), $n_A = n_B = n$.

Pooled proportion:
$$\hat{p} = \frac{TP_A + TP_B}{n_A + n_B}$$

Z-статистика:
$$z = \frac{p_B - p_A}{\sqrt{\hat{p}(1-\hat{p})\left(\frac{1}{n_A}+\frac{1}{n_B}\right)}}$$

### Доверительные интервалы

95% ДИ для разности пропорций $(p_B - p_A)$:

$$\Delta \pm 1.96 \cdot \sqrt{\frac{p_A(1-p_A)}{n_A} + \frac{p_B(1-p_B)}{n_B}}$$

### Критерий успешности

Тест считается успешным (v2 принимается в production), если выполнены **все** условия:

1. p-value < 0.05 (статистически значимое улучшение).
2. Нижняя граница 95% ДИ для $(F1_B - F1_A) > 0$ (практическая значимость).
3. Recall v2 ≥ Recall v1 (не хуже по ключевой бизнес-метрике).
4. Approval Rate v2 ≥ Approval Rate v1 при одинаковом пороге риска.

## 5. Практическая демонстрация

Обе версии модели уже доступны через API:

```bash
# Контрольная группа (v1)
curl -X POST "http://localhost:8080/predict?version=v1" \
  -H "Content-Type: application/json" -d '{...}'
# → {"model_version": "model_v1", "prediction": 1, "probability": 0.775}

# Тестовая группа (v2)
curl -X POST "http://localhost:8080/predict?version=v2" \
  -H "Content-Type: application/json" -d '{...}'
# → {"model_version": "model_v2", "prediction": 1, "probability": 0.84}
```

Сравнение метрик на hold-out выборке (test set, 6 000 строк):

| Метрика | v1 (LR) | v2 (RF) | Δ |
|---|---|---|---|
| ROC-AUC | 0.708 | 0.757 | +0.049 |
| F1 (дефолт) | 0.46 | 0.45 | −0.01 |
| Recall (дефолт) | 0.62 | 0.35 | −0.27 |
| Precision (дефолт) | 0.37 | 0.65 | +0.28 |
| Accuracy | 0.68 | 0.81 | +0.13 |

**Интерпретация.** v2 значительно точнее (Precision 0.65 vs 0.37) — меньше ложных тревог. v1 ловит больше реальных дефолтов (Recall 0.62 vs 0.35). Выбор финальной модели зависит от бизнес-приоритета: если критичнее *не пропускать* дефолты — оставить v1; если важнее *не отказывать* хорошим клиентам — выбрать v2. A/B-тест на реальном трафике прояснит, какой компромисс ведёт к меньшим финансовым потерям.

## 6. Влияние на архитектуру сервиса

Текущий монолитный сервис уже готов к A/B-тесту: параметр `?version` позволяет явно выбрать модель. В production схема дополняется:

```
Load Balancer / API Gateway
    ├── 50% трафика → /predict?version=v1  (контроль)
    └── 50% трафика → /predict?version=v2  (тест)
                            ↓
                    Логи → ELK / Loki
                            ↓
                    Анализ метрик → Grafana / Jupyter
```

После завершения теста маршрутизатор переключает 100% трафика на победившую версию без изменения кода сервиса.
